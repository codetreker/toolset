# 委派，不要亲自干

## 核心原则

你的主 session 是**指挥台**，不是工位。

- **你（主 session）**：接收消息、做决策、派任务、汇报结果
- **subagent**：具体执行——写代码、改配置、分析日志、跑长命令、读大量文件

## 什么时候 spawn subagent

| 场景 | 自己做 | spawn subagent |
|------|--------|---------------|
| 回复一句话 | ✅ | |
| 读一个短文件 | ✅ | |
| 改一两行配置 | ✅ | |
| 读多个文件分析 | | ✅ |
| 写代码 / 大量修改 | | ✅ |
| 跑测试 / 编译 | | ✅ |
| git 操作（add/commit/push） | ✅ | |
| 复杂的排查诊断 | | ✅ |
| 任何预计 > 1 分钟的操作 | | ✅ |

## 怎么派 subagent

好的派工：
- 明确目标："把 XXX 文件里的 A 改成 B，跑测试确认通过"
- 明确验收标准："测试全过、lint 通过、commit message 包含 issue 号"
- 明确约束："不要改 XXX 以外的文件"、"用 HTTPS 不用 SSH"

坏的派工：
- "去把这个搞好"（目标不明确）
- "参考 XXX 做一下"（没有验收标准）
- 一次塞太多不相关的事（应该拆成多个 subagent）

## 铁律

1. **人类消息 > 一切任务** — 收到人类消息立刻响应，不要因为 subagent 在跑就不理人
2. **主 session 不阻塞** — spawn 完 subagent 就继续待命，不要同步等
3. **subagent 完成后汇报** — 结果到了转达给相关人，不要让结果沉默
4. **大文件用 exec + heredoc** — subagent 写大文件容易因 JSON 截断失败，大文件直接在主 session 用 heredoc 写更可靠

## 自检

> **如果你发现自己在连续 read 文件、连续 exec 命令、或者一个操作做了好几分钟还没完——你就在当工兵了。停下来，spawn subagent。**

## 每个角色的应用

| 角色 | 自己做 | 委派 subagent |
|------|--------|--------------|
| 总管 | 接消息、做决策、派任务、主持站会 | 检查频道、分析日志、批量操作 |
| 架构 | 给 review 结论、回答技术问题 | 读大量代码做 review、写设计文档 |
| 开发 | 确认任务、汇报进度 | 写代码、跑测试、提 PR |
| 产品 | 回答需求问题、参与讨论 | 写 PRD、整理站会纪要 |
| QA | 给验收结论 | 跑测试套件、做 e2e 验证 |
