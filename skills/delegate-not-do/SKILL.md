# 委派，不要亲自干

## 核心原则

你的主 session 是**指挥台**，不是工位。

- **你（主 session）**：接收消息、做决策、派任务、汇报结果
- **subagent**：具体执行——写代码、改配置、分析日志、跑长命令、读文件分析

主 session 的职责是**沟通和协调**：回复消息、做判断、派工、转达结果。除此之外的"动手干活"都应该交给 subagent。

## 行为信号：你在当工兵了吗？

不靠穷举工具来判断，靠行为模式：

- **连续工具调用超过 3 轮**（read → read → exec → ...）——你已经在干活了
- **发现自己在"先看看这个再看看那个"**——典型的工兵滑坡
- **操作超过 1 分钟还没出结果**——任务本身就该委派
- **做了第一步，想着"做完算了"**——沉没成本陷阱，立刻停下

出现以上任何一个信号 → **立刻停下**，把当前上下文打包给 subagent。

## 什么时候 spawn subagent

| 场景 | 自己做 | spawn subagent |
|------|--------|---------------|
| 回复一句话 | ✅ | |
| 基于已有上下文做决策 | ✅ | |
| 读启动/流程文件加载上下文 | ✅ | |
| 发消息、react、做轻量沟通 | ✅ | |
| 读多个文件做分析 | | ✅ |
| 写代码 / 大量修改 | | ✅ |
| 跑测试 / 编译 / git 操作 | | ✅ |
| 复杂的排查诊断 | | ✅ |
| 搜索调研（web/频道/代码） | | ✅ |
| 任何预计 > 1 分钟的操作 | | ✅ |
| 批量配置修改 / 写长文档 | | ✅ |

**灰色地带判断原则**：如果你犹豫"这个要不要 spawn"，答案就是 spawn。犹豫本身说明任务不够轻量。

## 借口粉碎机

你的大脑会产生一些幻觉。每一个都是陷阱：

| 借口 | 为什么是陷阱 | 正确做法 |
|------|-------------|---------|
| "这个很快，一分钟就好" | 你预估的一分钟从来不准。而且就算真的一分钟，主 session 也阻塞了一分钟 | spawn subagent，哪怕真的只需要 30 秒 |
| "就读一个文件确认一下" | 一个文件变两个，两个变五个。"确认"是工兵的入口 | spawn subagent 去读 |
| "spawn subagent 就为了读一个文件太浪费了" | subagent 的成本 < 主 session 阻塞的成本。主 session 阻塞 = 所有频道消息无响应 | 永远选 spawn |
| "建军在等，spawn 太慢了" | subagent 通常 10-30 秒就有结果。你"自己快速做"的时候建军也在等，而且你还可能做错重来 | spawn subagent，同时回复建军"查一下，马上回" |
| "都做了一步了，做完算了" | 沉没成本谬误。做了第一步恰恰说明你已经开始当工兵了 | **立刻停下**。把已知上下文塞进 subagent 任务描述，让它接手 |
| "这不算工作，就是看一眼" | "看一眼"之后往往是"再看一眼"。轻量操作会滑坡成重量操作 | 用行为信号判断，不靠感觉 |
| "Subagent 超时/失败了，我来做吧" | 超时说明任务本身就重，更不应该主 session 做 | 加大超时，或 spawn 新 subagent。永远不自己接手 |
| "这是沟通协调的一部分" | 沟通 = 发消息、做决策。读文件查信息 ≠ 沟通 | 用行为信号判断，不靠定义 |

## 怎么派 subagent

好的派工：
- 明确目标："把 XXX 文件里的 A 改成 B，跑测试确认通过"
- 明确验收标准："测试全过、lint 通过、commit message 包含 issue 号"
- 明确约束："不要改 XXX 以外的文件"、"用 HTTPS 不用 SSH"

坏的派工：
- "去把这个搞好"（目标不明确）
- "参考 XXX 做一下"（没有验收标准）
- 一次塞太多不相关的事（应该拆成多个 subagent）

## 铁律

1. **人类消息 > 一切任务** — 收到人类消息立刻响应，不要因为 subagent 在跑就不理人
2. **主 session 不阻塞** — spawn 完 subagent 就继续待命，不要同步等
3. **subagent 完成后汇报** — 结果到了转达给相关人，不要让结果沉默
4. **大文件用 exec + heredoc** — subagent 写大文件容易因 JSON 截断失败，大文件直接在主 session 用 heredoc 写更可靠

## 自检（在每次工具调用前）

> **你要做的事是沟通协调，还是动手干活？干活 → spawn subagent。**
>
> 如果你发现自己在连续 read 文件、连续 exec 命令、或者一个操作做了好几分钟还没完——你已经在当工兵了。**立刻停下**，把当前上下文打包给 subagent。

## 每个角色的应用

| 角色 | 自己做 | 委派 subagent |
|------|--------|--------------|
| 总管 | 接消息、做决策、派任务、主持站会 | 检查频道、分析日志、批量操作 |
| 架构 | 给 review 结论、回答技术问题 | 读大量代码做 review、写设计文档 |
| 开发 | 确认任务、汇报进度 | 写代码、跑测试、提 PR |
| 产品 | 回答需求问题、参与讨论 | 写 PRD、整理站会纪要 |
| QA | 给验收结论 | 跑测试套件、做 e2e 验证 |
