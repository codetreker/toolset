# Something Worth Wiki

## 什么值得写

### ✅ 写

| 类型 | 示例 |
|------|------|
| **隐蔽的坑** | 一个配置项的行为跟文档说的不一样，排查了半天才发现 |
| **反直觉的行为** | 某个 API 在特定条件下静默失败，不报错 |
| **可复用的方法** | 总结出了一套调试某类问题的标准步骤 |
| **架构决策的 why** | 为什么选了方案 A 不选方案 B，当时的 trade-off 是什么 |
| **踩过的坑 + 解法** | 问题是什么、根因是什么、怎么解决的、怎么避免再犯 |
| **工具的正确用法** | 官方文档没写清楚或容易误解的用法 |

### ❌ 不写

| 类型 | 理由 |
|------|------|
| 日常 bug fix | 修了就完了，不需要沉淀 |
| 常规配置变更 | 改了个端口号不值得一篇文章 |
| 官方文档已经写清楚的 | 不重复造轮子，贴个链接就行 |
| 流水账 | "今天做了 X、Y、Z"是日志不是知识 |
| 还没验证的猜测 | 等确认了再写 |

### 判断口诀

> **三个月后的你，如果再遇到同样的问题，这篇文章能让你少走弯路吗？**
>
> - 能 → 写
> - 不能 → 不写
> - 不确定 → 大概率不值得写

## 怎么写

### 通用格式

每篇 wiki 必须符合 MkDocs Material 格式：

```markdown
# 标题：一句话说清楚这篇在讲什么

!!! info "作者"
    xxx | 日期

正文...

!!! warning "注意"
    重要提醒用 admonition

!!! tip "技巧"
    实用建议用 tip
```

**MkDocs 格式要点：**
- 用 `#` 层级组织标题（`#` → `##` → `###`，不跳级）
- 用 admonition（`!!! info/warning/tip/danger`）高亮重点信息
- 代码块标注语言：` ```bash ` / ` ```json ` / ` ```python `
- 表格、列表正常用 Markdown 语法
- 不要用 HTML 标签

正文结构**根据内容类型自由组织**，没有固定模板。不同类型的知识适合不同的写法：

#### 踩坑型
问题 → 根因 → 解法 → 怎么避免再犯

#### 方法型
场景 → 做法 → 关键步骤/命令 → 注意事项

#### 概念型
是什么 → 为什么重要 → 怎么用 → 常见误解

#### 决策型
背景 → 选项对比 → 选了什么 → 为什么

#### 工具型
用途 → 安装/配置 → 常用命令 → 踩坑点

**不要硬套模板。** 哪种结构能最快让读者理解就用哪种。

### 写作原则

1. **写结论，不写过程**
   - ❌ "先试了 A，不行，又试了 B，还是不行，最后试了 C 才行"
   - ✅ "解法是 C，因为 Z"（过程可以简要提，但结论在前）

2. **写给失忆的人**
   - 假设读者什么上下文都没有
   - 不要写"如前文所述"——没有前文
   - 关键术语第一次出现时解释

3. **附上能复制粘贴的东西**
   - 关键命令、配置片段、代码示例
   - 不要只说"修改配置文件"——写清楚改哪个文件的哪个字段改成什么

4. **一篇一个主题**
   - 不要把三个不相关的坑塞进一篇
   - 如果内容太多，拆成多篇

5. **标题要能搜到**
   - ❌ "踩坑记录 #42"
   - ✅ "Discord allowBots 配置导致 bot-to-bot 消息静默丢失"

6. **不要硬凑字数**
   - 三句话能说清楚的不要写三段
   - 短而准 > 长而空

## Wiki 仓库规范

### 目录结构

```
docs/
├── infra/          # 基础设施（部署、运维、容器）
├── openclaw/       # OpenClaw 相关经验
├── tools/          # 工具使用技巧
├── dev/            # 开发经验（Git、代码、调试）
└── concepts/       # 概念理解（架构、设计模式）
```

### 文件命名

- 用英文小写 + 短横线：`discord-multi-agent-config.md`
- 不用日期前缀（wiki 不是日志）
- 名字要能看出内容

### 提交流程

1. **先查重**：浏览 wiki 目录，看有没有已存在的相关文章。如果有，在原文基础上增补，不要另起一篇重复的
2. 在对应目录创建 `.md` 文件（确认没有重复后）
3. 按上面的模板写
4. 检查隐私（不含真实 ID、token、密钥等）
5. 更新 `mkdocs.yml` 的 nav 加入新页面（新文章才需要）
6. commit + push

### 隐私红线

这是公开仓库。**任何真实的个人信息、Discord ID、token、API key、内部 URL 一律用占位符替代。**

## 什么时候触发

- **站会结尾**：总管问"今天有没有值得写 wiki 的？"——这时候用这个 skill 判断
- **踩坑之后**：刚解决了一个折腾很久的问题——趁热写
- **Review 时发现通用教训**：这个错误别人也会犯——值得写
- **有人问了一个好问题**：答案值得沉淀——写下来以后不用再答
