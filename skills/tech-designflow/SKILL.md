# Tech Design Flow

技术设计文档的标准流程。从调研到审批，每一步都有明确产出和退出条件。

## 前置条件

- 有明确的技术问题或需求（新模块、架构变更、存储选型等）
- 问题复杂度值得写设计文档（5 分钟能搞定的不需要）
- 已有项目频道和任务板
- **遵循 git-workflow skill**：所有文件写操作在任务分支的 `.worktrees/` 里进行
- **一任务一分支**：设计文档写在任务分支上，不单独开分支/PR，待整个任务完成后随代码一起合并

## 流程总览

```
调研 → 写设计文档 → Review Round 1 → 修订 → Review Round 2（可选）→ 提交审批
```

最多 3 轮 Review。3 轮后仍有 P0 → 升级到人类决策，不再无限循环。

---

## Phase 1: 调研

### 目标

收集足够信息支撑技术决策。不是写论文，是回答"我们该怎么做，为什么"。

### 步骤

1. **定义调研问题**：明确要回答什么（例如："GSE vs ICU 分词，哪个适合我们的多语言索引场景？"）
2. **派 2 个 Claude Code 并行调研**：
   - 每个给不同侧重点（例如：性能、兼容性、社区生态）
   - 明确产出格式：结论 + 依据 + 风险
   - 设置合理 scope，避免发散
3. **汇总调研结论**：
   - 合并去重，标注共识和分歧
   - 对分歧点做进一步验证或标记为开放问题
   - 产出调研摘要（< 2 页）

### 调研 Prompt 模板

```
调研主题：{topic}
侧重方向：{focus}
需要回答的问题：
1. {question_1}
2. {question_2}
3. {question_3}

产出格式：
- 结论（1-2 句话）
- 关键依据（带来源链接）
- 风险和注意事项
- 推荐方案及理由
```

### 退出条件

- 核心问题有明确结论
- 备选方案至少 2 个，各有优劣分析
- 无重大未解决分歧（或已标记为开放问题）

---

## Phase 2: 写设计文档

### 目标

基于调研结论，产出完整的技术设计文档。

**写设计文档用 coding agent**（Claude Code / Codex），调用方式见 `using-claude-code` / `using-codex` skill。

**注意**：`--print` 模式只输出文本，不能写文件。要写文件必须用 shell 重定向 `> file.md`，或者用 ACP 模式让 agent 直接写文件。

### 设计文档结构

```markdown
# {标题}

## 背景与问题
为什么要做这个？解决什么问题？

## 目标
做完之后世界变成什么样？（可量化的验收标准）

## UI 设计稿
如果涉及 UI 改动，链接到 `docs/ui/` 下的线框图文件。
例如：`- [XXX 线框图](../ui/xxx.md)`

## 方案设计
### 整体架构
### 核心流程
### 数据结构 / API 设计
### 错误处理

## 备选方案
### 方案 B：{名称}
- 优点
- 缺点
- 为什么没选

## 测试策略
- 单测覆盖哪些
- 集成测试覆盖哪些
- 边界和异常场景

## Task Breakdown
按 DDD skill 的 Task Breakdown 规范拆分

## 风险与开放问题
- 已知风险及缓解措施
- 待确认的开放问题

## 参考资料
- 调研文档链接
- 相关论文 / 文档链接
```

### 写作原则

- **遵循 git-workflow**：先创建 worktree，在 worktree 里写文档，不在主目录写文件
- **设计 ≠ PRD 换格式**：PRD 回答「测/做什么」，设计回答「怎么做」。如果设计文档只是把 PRD 的场景展开成表格，没有新增技术决策，那就是在浪费时间。设计必须包含：技术选型理由、实现方案（代码级别）、数据结构、依赖关系、mock/stub 策略、测试隔离方案等 PRD 里没有的内容。
- **框架 + 具体用例缺一不可**：只有架构框架没有具体用例 = 空中楼阁。设计文档的每个模块必须展开到可执行粒度。方案设计要有：具体的 API 请求/响应示例、数据结构定义（字段+类型+约束）、核心函数签名、数据流走查（A 调 B → B 查 C → 返回 D）、配置示例。测试设计要有：具体的 test case（setup/action/assertion）、mock/stub 方案、隔离策略。开发拿到设计文档应该能直接写代码，不需要自己再设计细节。（B24 教训：v2 只抄 PRD，v3 只有框架，到 v4 才补上具体用例。）
- **涉及 UI 必须链接设计稿**：如果设计涉及 UI 改动，在“目标”后加“UI 设计稿”章节，链接到 `docs/ui/` 下的线框图文件。Dev 开发时能直接参考。
- **够用就行**：不是写论文，是写施工图。开发能看懂、能动手就够。
- **备选方案必须写**：将来需求变了可以回头看当初为什么没选。
- **测试策略不能空**：QA 需要这个来做验收。
- **Task Breakdown 必须有**：没有拆分的设计文档 = 没有设计文档。

### 退出条件

- 文档覆盖上述所有章节
- Task Breakdown 已拆分到可执行粒度
- 无 TBD / TODO 占位符（开放问题归到"风险与开放问题"章节）

## Phase 3: Review

### 目标

发现设计缺陷，在写代码之前修复。

### Review 流程

1. **派 2 个 Reviewer 并行 review**（用不同的 coding agent，如 Claude Code + Codex）：
   - 调用方式见 `using-claude-code` / `using-codex` skill
   - **Timeout 至少 30 分钟**
   - 两个并行不阻塞
2. **收集 review 意见**（`process log` 读取输出）
3. **按严重度分级**（见下方）
4. **修订文档**
5. **如有 P0 修改，进入下一轮 review**

### Review Prompt 模板

```
请 review 以下技术设计文档。

Review 要求：
1. 对照"目标"检查方案是否能达成
2. 检查边界条件和错误处理是否完整
3. 检查备选方案分析是否合理
4. 检查 Task Breakdown 粒度是否合适
5. 标注每个问题的严重度（P0/P1/P2）

严重度定义：
- P0（必须修）：逻辑错误、安全漏洞、架构缺陷、会导致返工
- P1（应该修）：边界遗漏、命名不清、测试覆盖不足
- P2（建议改）：风格偏好、文档措辞、非关键优化

{设计文档全文}
```

### 严重度分级

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **P0** | 逻辑错误、安全漏洞、架构缺陷、会导致返工 | 必须修，修完重新 review |
| **P1 | 边界遗漏、命名不清、测试覆盖不足 | 必须修，当轮修完 |
| **P2** | 风格偏好、文档措辞、非关键优化 | 酌情采纳，不阻塞流程 |

### 修订规则

- **P0 问题全部修复后**，进入下一轮 review（仅 review 修改部分 + 受影响部分）
- **只有 P1/P2**，当轮修完，不需要额外 review 轮次
- **没有任何问题**，直接进入审批

### Review 轮次限制

- **最多 3 轮 review**
- 3 轮后仍有 P0 → 说明方案本身可能有根本问题，升级到人类决策
- 正常情况 1-2 轮足够

### 退出条件

- 所有 P0 已修复并验证
- P1 已修复或明确标注为已知限制
- Review 轮次 ≤ 3

---

## Phase 4: 提交审批

### 步骤

1. **推设计文档到仓库**（`docs/design/` 或项目约定目录）
2. **生成 review 链接**（使用 ready-for-review skill）
3. **在项目频道通知人类**，附上：
   - 设计文档链接
   - 一句话总结方案
   - 调研和 review 过程摘要（几轮 review、主要修改点）
4. **等待人类审批**
   - 审批通过 → 更新任务板，进入开发阶段
   - 审批有修改意见 → 回到 Phase 3 修订
   - 审批否决 → 记录原因，关闭或重新调研

### 退出条件

- 人类明确说"通过"/"approved"/"LGTM"
- 设计文档已在仓库中（不是只在聊天记录里）
- 任务板状态已更新

---

## 反模式（不要做）

- ❌ 调研只派 1 个 agent → 容易有盲区，至少 2 个并行
- ❌ Review 只用一个工具 → Claude Code + Codex 并行，不同模型视角互补
- ❌ 跳过 review 直接提交审批 → review 是质量保证，不是形式
- ❌ Review 超过 3 轮还在改 → 说明方案有根本问题，升级决策
- ❌ 设计文档只在聊天里讨论不落文件 → 口头设计 = 没有设计
- ❌ 修了 P0 不重新 review → P0 修改可能引入新问题
- ❌ Task Breakdown 写"实现功能 X" → 粒度不够，要拆到具体步骤
- ❌ 设计文档只有框架没有具体用例 → 方案设计缺 API 示例/数据流走查/schema 定义，测试设计缺具体 test case，都算不合格
- ❌ 设计文档 = PRD 场景列表 + 技术术语 → 设计必须包含 PRD 里没有的技术决策和实现细节

## 快速检查清单

开始前：
- [ ] 问题足够复杂，值得走设计流程？
- [ ] 项目频道和任务板已就绪？

调研后：
- [ ] 核心问题有结论？
- [ ] 至少 2 个备选方案？

设计文档写完后：
- [ ] 所有章节都填了？（无 TBD）
- [ ] Task Breakdown 到可执行粒度？

Review 后：
- [ ] 所有 P0 已修复？
- [ ] Review 轮次 ≤ 3？

提交审批前：
- [ ] 文档已推到仓库？
- [ ] 有 review 链接？
